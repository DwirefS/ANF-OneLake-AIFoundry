The Problem & The Solution

What This Solves

Enterprises want to build AI agents on their existing data. But today that means copying data into new stores, building brittle ETL pipelines, and re-platforming to formats AI services understand. This project eliminates all of that.

No Data Duplication

Data stays in Azure NetApp Files — the same NFS/SMB volumes your applications already use. OneLake virtualizes access through S3-compatible shortcuts. Zero copies, zero new storage silos, zero ETL pipelines to maintain.

Complete AI Agent Infrastructure

This is the base scenario — not a throwaway demo. Networking, storage, data gateway, OneLake integration, intelligent search indexing with Document Intelligence, GPT-4o model deployments, and a grounded agent. The full stack, wired end-to-end.

Deploy and Go

Run one script, wait ~25 minutes, open AI Foundry and start chatting. The automated deployment handles everything — infrastructure provisioning, data gateway setup, Fabric configuration, search indexing, and agent creation. Then customize from there.

Architecture

The Zero-Copy AI Data Path

Four layers, four Azure services. Data flows from enterprise storage to AI agent responses without ever leaving its authoritative location.

Layer 1 · Storage

Azure NetApp Files

Enterprise NFS/SMB volume with S3-compatible Object REST API. Your data stays here — governed, performant, and dual-protocol accessible.

Layer 2 · Integration

OneLake Shortcut

Microsoft Fabric virtualizes ANF data via S3-compatible shortcut through an On-Premises Data Gateway. No data movement. No copies.

Layer 3 · Indexing

Azure AI Search

OneLake indexer extracts, processes with Document Intelligence, chunks, and vectorizes content for hybrid retrieval.

Layer 4 · Intelligence

AI Foundry Agents

GPT-4o agents grounded on the vector index. Ask questions about your enterprise data, get answers with document citations.

Why Zero-Copy Matters

Enterprise data stays governed in Azure NetApp Files. No new storage silos. No brittle ingestion pipelines. The AI agent queries a pre-built vector index at runtime — it never touches the source storage. This is how production-grade RAG should work.

Automated Deployment

One Script. Full Stack. Ready to Go.

The deploy.ps1 orchestrator chains eight steps, passing outputs between them. Every service that was manually configured in the hands-on lab is automated via Bicep templates, Azure REST APIs, Fabric REST APIs, and AI Search REST APIs.

Steps 1-2

Infrastructure

Register providers. Deploy VNet, ANF volume, gateway VM, AI Search, AI Services, AI Foundry Hub & Project via Bicep.

Steps 3-5

Data & Gateway

Auto-generate TLS cert. Create S3 bucket + credentials. Upload test data. Install gateway silently on VM. Register with service principal.

Step 6

Fabric & OneLake

OAuth2 auth. Create workspace & lakehouse. Discover gateway. Create S3 connection. Create OneLake shortcut. All via Fabric REST API.

Steps 7-8

AI Search & Agent

Create data source, 5-skill skillset, vector index, indexer. Run indexer. Create grounded AI agent. Verify with test query.

How Every Manual Step Was Automated

Every portal click, every RDP session, every wizard step from the hands-on lab has been replaced:

Infrastructure as Code

Six Bicep modules deploy all Azure resources in one shot: VNet, ANF, gateway VM, AI Search, AI Services (with GPT-4o + embedding models), AI Foundry Hub + Project with all connections.

Auto-Generated TLS Certificate

No more manual openssl commands. The script auto-generates a self-signed cert for the ANF Object REST API bucket, or accepts a user-provided one.

Silent Gateway Install

No RDP required. The gateway is downloaded, installed silently, and registered via service principal — all through az vm run-command executed from your local machine.

Fabric via REST API

Workspace, lakehouse, S3 connection, and OneLake shortcut are created programmatically using OAuth2 client_credentials authentication. No Fabric portal needed.

AI Search via REST API

Data source, 5-skill skillset (with Document Intelligence), vector index, and indexer are all created and run via REST API. The index schema is externalized as a JSON config file.

Agent + Verification

The AI Foundry agent is created via REST API with AI Search tool binding, then a test query is automatically run to verify the entire pipeline returns grounded answers.

Build From Here

The Foundation is Ready. Now Customize.

This project gives you the complete base integration — all the hard plumbing between ANF, OneLake, AI Search, and AI Foundry is done. From here, you can extend in any direction:

Add Your Own Data

Replace the sample financial data with your own documents. The indexing pipeline handles PDFs, images, CSVs, and text files. Drop files on the ANF volume and re-run the indexer.

Add Custom Skills

Extend the AI Search skillset with custom skills — entity extraction, translation, classification, PII redaction — by adding to the skillset definition. The pipeline is modular.

Customize the Agent

Change the system prompt, add new tools (Code Interpreter, Azure Functions, Bing Search), adjust retrieval parameters, or create multiple specialized agents from the same index.

Multi-Source RAG

Add more OneLake shortcuts pointing to different ANF volumes, Azure Blob, ADLS Gen2, or other S3-compatible stores. AI Search can index from multiple data sources into a single index.

Production Hardening

Add private endpoints, customer-managed encryption keys, Azure Monitor diagnostics, and managed identity for all service-to-service authentication. The Bicep modules are ready to extend.

Integrate Into Existing Apps

The AI Foundry agent exposes a standard OpenAI-compatible API. Embed it into your existing applications, Teams bots, Power Apps, or custom web frontends.
