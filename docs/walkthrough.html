Presenter Guide

Walkthrough & Talk Track

This narrative explains why each step exists and what to emphasize during live workshops or customer discussions. Follow alongside the Lab Guide.

1. Validate Azure Subscription Requirements

What we're doing

Making sure the Azure subscription is fully enabled to support storage, analytics, and AI services before we start building anything.

Why this matters

This workshop spans multiple Azure services—Azure NetApp Files, Microsoft Fabric, Azure AI Search, and Azure AI Foundry. If required resource providers or permissions are missing, later steps can fail in ways that are hard to diagnose.

What to emphasize:
  • This is standard Azure hygiene, not something unique to this solution.
  • We're front-loading this to avoid surprises later.
  • Once this is done, the rest of the workshop focuses on solution flow, not subscription troubleshooting.

2. Prepare Azure NetApp Files (Storage)

2.1–2.2 Create NetApp Account, Pool, and Volume

We're creating an Azure NetApp Files volume that will act as the authoritative system of record for enterprise file data. Azure NetApp Files provides enterprise file semantics (NFS/SMB), high performance, low latency, governance, and durability.

This data is not being migrated or transformed—it remains in place throughout the workshop.

Key Point

Once this volume exists, Azure NetApp Files is fully configured. Everything that follows is about how other services consume this data without copying it.

2.3 Enable Object (S3-Compatible) Access

We're enabling the Azure NetApp Files Object REST API so the same file data can be accessed through an S3-compatible interface.

What to emphasize:
  • This is still the same data.
  • No ingestion, duplication, or ETL is happening.
  • We're exposing another access path, not creating another copy.

2.4 Upload Lab Data

We're placing sample unstructured data into Azure NetApp Files so we have real content to work with. This data will later be virtualized into OneLake, indexed by Azure AI Search, and used to ground an AI agent. At no point will it be copied into a separate data store.

3. Create a OneLake Shortcut to Azure NetApp Files

3.1 Deploy Data Gateway

We're deploying a Microsoft Fabric Standard Data Gateway. This gateway is required by Microsoft Fabric, not by Azure NetApp Files. Fabric needs a secure way to reach private, VNet-isolated, S3-compatible endpoints.

Important clarification: Azure NetApp Files does not require a gateway. The gateway exists purely to allow Fabric to access private storage securely.

3.2 Create Fabric Workspace

We're creating a Microsoft Fabric workspace that will own the Lakehouse, the OneLake shortcut, and permissions used later by Azure AI Search.

3.3 Create Connection and OneLake Shortcut

We're creating a OneLake shortcut that points to the Azure NetApp Files bucket using S3-compatible access.

Core Architectural Moment

This is the pivotal step of the workshop. OneLake now has a logical view of data that physically lives in Azure NetApp Files. No data is copied into Fabric. Any Fabric-connected service can now "see" the data. Shortcuts are virtualization, not ingestion. This is how we avoid data sprawl. The data remains governed by its original storage system.

4. Index OneLake Data Using Azure AI Search

4.1 Create Azure AI Search Service

Azure AI Search indexes unstructured content, creates embeddings, and enables fast, relevant retrieval for AI applications. This is what bridges data and AI.

4.2 Assign Permissions

Azure AI Search must be able to read OneLake data and respect Fabric security boundaries. Without this permission, indexing will silently fail.

4.3 Import and Vectorize Data

This step turns raw files into searchable content and vector representations usable by AI agents.

Workshop guardrails to mention verbally:
  • We index files under the Lakehouse Files/ path.
  • We avoid sensitivity-labeled content for this workshop.
  • We use reasonably sized text-based documents.

5. Connect OneLake to Azure AI Foundry

5.1 Create Azure AI Services Resource and Deploy Models

We're provisioning the AI runtime and deploying the models used by the agent. This ensures predictable behavior during the workshop, explicit control over models and region, and no hidden dependencies.

What to emphasize: In production, model choices may vary. For the workshop, we standardize this to keep things smooth.

5.2 Assign User Permissions

Without this, the Foundry UI may load but inference will fail. This step makes sure the workshop can proceed without RBAC friction and attendees can focus on the experience, not access errors.

5.3 Create Foundry Hub and Project

The project is where agents are defined, knowledge sources are attached, and evaluation and iteration happen.

6. Run and Validate a Grounded Agent

We're creating an AI agent and grounding it on the indexed enterprise data.

The Payoff

The agent answers questions. Responses are grounded in real enterprise files. Data stays in Azure NetApp Files the entire time. This is not a demo on sample data in a hidden store. This is AI operating directly on governed enterprise data—without copying it.

Closing Message

What we've demonstrated is a repeatable pattern:

  • Azure NetApp Files as the system of record
  • OneLake as a virtualization layer
  • Azure AI Search as the retrieval substrate
  • Azure AI Foundry as the AI execution layer

This pattern scales from workshops to real enterprise deployments, while minimizing risk, duplication, and operational overhead.

How to Use This

  1. Distribute: Send the lab guide and test data to your workshop participants.
  2. Pre-Flight: Ensure subscriptions are whitelisted for the Azure NetApp Files Object REST API Access preview.
  3. Run: Follow the guide step-by-step. Section 6 provides the "wow" moment for the demo.
